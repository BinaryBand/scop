# Audit Report

## 1. Internal issues within SCOP

**¬ß7.2 ‚Äî `PROCESS_LOG` required field inconsistency**

The MSGID table says `PROCESS_LOG` has required field `id` and no `Optional` fields. But the prose says "`PROCESS_LOG` carries its payload in `msg` only ‚Äî no separate `message` field." Both `id` and `msg` are required (the latter by the global wire format rule), so the table is fine, but the explanatory note is slightly misleading ‚Äî it implies `msg` is the *only* field, which undersells the requirement for `id`.

**¬ß9 ‚Äî Page Template slot map missing `intent` field routing**

The slot map table routes events by flag and MSGID, but `intent` is the actual routing discriminant used in ¬ß10. The table says "any" in the Flag column for the Page Chrome slot, which is underspecified. A reader following ¬ß9 alone might not know to set `intent: "query"` vs `intent: "action"` ‚Äî the template section doesn't mention it, and ¬ß7.1 defines it separately. A forward reference or a note here would prevent implementors missing it.

**¬ß8.1 ‚Äî `--list` contract uses OR without a tiebreaker rule**

The contract says TABLE or LIST depending on whether items have named fields. This is fine as English prose but not machine-enforceable ‚Äî there's no formal rule for when a producer must choose one over the other, nor does SCOP-M's `[room.list]` schema cover the scalar-list case. Minor, but worth a note.

## 2. Internal issues within SCOP-M

**¬ß4.6 ‚Äî `kind` field referenced but not defined in command table**

The `[[room.command]]` schema table (¬ß4.6) lists `name`, `exec`, `description`, and `navigates` ‚Äî but no `kind` field. Yet conformance rule #9 says: "Include `params` on every `[[room.command]]` with `kind = "action"`." A `kind` field for commands is implied but never defined anywhere in the schema. This is a dangling reference.

**¬ß6 ‚Äî Semantic equivalence table references `room.command.kind`**

The equivalence table maps `room.command.exec`, `.description`, `.kind` to the runtime `LIST_APPEND.value` fields. But again, `kind` on a command is undefined in the schema. The SCOP `--help` contract (¬ß8.1) defines `params[].type` but there's no `kind` at the command level in the runtime either. This appears to be a leftover from a prior draft.

**¬ß4.7 ‚Äî Complete example missing `log` room**

The root room in the complete example (¬ß7) declares four commands including `Log` with `navigates = "log"`, but there is no `[[room]]` entry with `id = "log"`. This is a conformance violation by the spec's own rule #10: "Include a `[[room]]` entry for every room the application emits events in."

**¬ß4.7 ‚Äî `restore` room example missing a required field**

The `restore` room's command declares a positional param with `name = "name"` but no `description` field. This is technically optional per the schema (¬ß4.7 marks it without a ‚úì), so not a violation ‚Äî but worth noting given it's the showcase example.

## 3. Internal issues within Sempy Architecture

**MSGIDs table ‚Äî `PROCESS_LOG` wrong required field**

The architecture doc's MSGIDs table lists `PROCESS_LOG` with required field `id`, `message`. But SCOP ¬ß7.2 explicitly says `PROCESS_LOG` carries its payload in `msg` only, with **no separate `message` field**. The architecture uses `message`; SCOP uses `msg`. This is a direct contradiction with the spec it claims to implement.

**`ResolvedResult.data` must be a `PAGE_END` message**

This is stated as a rule but not explained or cross-referenced. `PAGE_END` in SCOP is a stream terminator with no payload fields beyond `room` and `msg`. It's unclear why the resolved result datum must specifically be `PAGE_END` rather than a `PROCESS_END` or similar ‚Äî the architectural rationale is missing, and there's no enforcement mechanism cited.

**`dispatch()` code sample ‚Äî `spawn` signature mismatch**

The `RuntimePort` class diagram shows `spawn(coro: Coroutine, stream: StreamPort)`, but the `dispatch()` code sample calls `self._runtime.spawn(app.run(args, stream), stream)`. The second argument `stream` in the call would be redundant if `spawn` is meant to use the stream it already created one line earlier. Either the signature is wrong (stream not needed as a parameter since the adapter owns it) or the design intent isn't explained clearly.

## 4. Cross-document inconsistencies

**SCOP ¬ß7 vs Architecture MSGIDs table ‚Äî field name `message` vs `msg`**

Already noted above, but to be explicit as a cross-doc issue: Architecture says `PROCESS_LOG` requires `message`; SCOP says the payload lives in `msg` (the globally required field). `msg` is the correct field per SCOP ¬ß5's wire format rule.

**SCOP ¬ß5 requires `room` in every event; Architecture wire format examples comply, but the class diagram doesn't model it**

`SyslogMessage` in the class diagram has no fields shown. This isn't wrong exactly ‚Äî it's a partial diagram ‚Äî but `room` being a required field everywhere is important enough that omitting it from the model entirely could mislead an implementor about what `SyslogMessage` must carry.

**SCOP-M ¬ß4.6 `navigates` routing rule vs SCOP ¬ß6 room derivation**

SCOP-M says `navigates` must equal the room `id` SCOP will derive at runtime. In the complete example, `exec = "snap"` and `navigates = "snap"` ‚Äî correct. But SCOP ¬ß6's derivation rule joins *subcommand tokens* ‚Äî so if the app is invoked as `ourapp snap`, the room is `"snap"`. If the exec token is a subcommand of a subcommand (e.g. `diff compare`), then `navigates` should be `"diff/compare"`, not `"compare"`. The SCOP-M spec states this correctly in the routing rule note, but the complete example only shows single-depth cases. Not a bug, but worth a test case.

**SCOP ¬ß8.1 `--help` `LIST_APPEND.value.params` vs SCOP-M ¬ß6 equivalence table**

SCOP says `--help` items MAY include a `params` array and producers with a SCOP-M manifest SHOULD include them. SCOP-M's equivalence table says `room.command.param.*` maps to `LIST_APPEND.value.params`. This is consistent. However, SCOP-M conformance rule #9 restricts the SHOULD to commands with `kind = "action"` ‚Äî but since `kind` on commands is undefined (issue above), this rule is unenforceable as written.
