# The Syslog Protocol

## Abstract

This document describes the syslog protocol, which is used to convey event notification messages. This protocol utilizes a layered architecture, which allows the use of any number of transport protocols for transmission of syslog messages. It also provides a message format that allows vendor-specific extensions to be provided in a structured way.

## 1. Introduction

This document describes a layered architecture for syslog. The goal of this architecture is to separate message content from message transport while enabling easy extensibility for each layer.

This document describes the standard format for syslog messages and outlines the concept of transport mappings. It also describes structured data elements, which can be used to transmit easily parseable, structured information, and allows for vendor extensions.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC2119] and [RFC8174].

---

## 2. Terminology

- **SYSLOG-MSG** - a complete syslog message following the RFC 5424 ABNF definition.
- **HEADER** - the structured prefix containing PRI, VERSION, TIMESTAMP, HOSTNAME, APP-NAME, PROCID, and MSGID.
- **PRI** - the Priority value, encoding Facility and Severity as (Facility × 8) + Severity.
- **Facility** - a numeric code (0–23) indicating the type of process generating the message.
- **Severity** - an importance level (0–7) from EMERG to DEBUG, as defined in this document.
- **STRUCTURED-DATA** - optional field providing machine-parseable key-value pairs.
- **SD-ELEMENT** - a named collection of PARAM-NAME / PARAM-VALUE pairs within STRUCTURED-DATA.
- **MSG** - the free-form UTF-8 log line; mapped to the `msg` field in SCOP.

---

## 3. Design Principles

| Principle             | Rule                                                                        |
| --------------------- | --------------------------------------------------------------------------- |
| Layered               | Separates message content from transport; any transport mapping may be used |
| Extensible            | Vendor-specific extensions are supported via STRUCTURED-DATA SD-IDs         |
| Structured            | SD-ELEMENTs provide machine-parseable data alongside free-form MSG          |
| Transport-independent | Message format is defined independently of any specific transport protocol  |
| Backwards-compatible  | Inspired by BSD syslog (RFC 3164) while providing a Standards-Track format  |

---

## 4. Message Format

A syslog message has the following ABNF definition (simplified):

```abnf
SYSLOG-MSG = HEADER SP STRUCTURED-DATA [SP MSG]
HEADER     = PRI VERSION SP TIMESTAMP SP HOSTNAME SP APP-NAME SP PROCID SP MSGID
```

Every octet of the MSG part MUST be a valid UTF-8 character. Syslog messages MUST NOT exceed 480 octets; they SHOULD NOT exceed 2048 octets.

---

## 5. PRI — Facility and Severity

The PRI value is composed as `(Facility × 8) + Severity`, encoded as a decimal integer between angle brackets (e.g. `<134>`).

**Facility codes** (0–23):

| Code  | Keyword  | Description         |
| ----- | -------- | ------------------- |
| 0     | kern     | Kernel messages     |
| 1     | user     | User-level messages |
| 16    | local0   | Local use 0         |
| 17–23 | local1–7 | Local use 1–7       |

**Severity values** (0–7):

| Value | Keyword | Description                      |
| ----- | ------- | -------------------------------- |
| 0     | EMERG   | System is unusable               |
| 1     | ALERT   | Action must be taken immediately |
| 2     | CRIT    | Critical conditions              |
| 3     | ERR     | Error conditions                 |
| 4     | WARNING | Warning conditions               |
| 5     | NOTICE  | Normal but significant condition |
| 6     | INFO    | Informational messages           |
| 7     | DEBUG   | Debug-level messages             |

SCOP uses severity only (0–7) as the `pri` wire value, fixing the facility at 0. See SCOP §4.2.1.

---

## 6. HEADER Fields

The HEADER consists of the following fields:

| Field     | Key     | Required | Description                                       |
| --------- | ------- | -------- | ------------------------------------------------- |
| PRI       | `pri`   | ✓        | Facility × 8 + Severity                           |
| VERSION   | —       | ✓        | Always `1` for RFC 5424                           |
| TIMESTAMP | `ts`    | —        | ISO 8601 (e.g. `2003-10-11T22:14:15.003Z`)        |
| HOSTNAME  | —       | —        | FQDN, IP address, or hostname                     |
| APP-NAME  | `app`   | —        | Device or application name                        |
| PROCID    | `pid`   | —        | Process name or identifier                        |
| MSGID     | `msgid` | ✓        | Message type identifier (SCOP uses as event type) |

SCOP requires `pri`, `msgid`, and `msg` on every event. `ts`, `app`, and `pid` are optional.

---

## 7. STRUCTURED-DATA

STRUCTURED-DATA provides a mechanism to express information in a well-defined, easily parseable format. It consists of zero or more SD-ELEMENTs, each containing an SD-ID and zero or more SD-PARAMs.

```abnf
STRUCTURED-DATA = NILVALUE / 1*SD-ELEMENT
SD-ELEMENT      = "[" SD-ID *(SP PARAM-NAME "=" %d34 PARAM-VALUE %d34) "]"
```

SCOP does not use STRUCTURED-DATA; all structured content is carried in the MSGID-specific JSON fields of the event body.

---

## 8. MSG

The MSG part contains the free-form log message. It MUST be a valid UTF-8 string. A BOM (`%xEF.BB.BF`) MAY be prepended to indicate UTF-8 encoding.

In SCOP, the `msg` field maps to MSG and carries a complete, human-readable single line. It MUST NOT contain newline characters and MUST NOT verbatim duplicate any other field in the event.

---

## 9. Security Considerations

**Plaintext transport.** Syslog has traditionally been transported over plaintext UDP, meaning messages are visible to any network intermediary. Deployments SHOULD use an encrypted transport (e.g. TLS) to protect message confidentiality and integrity.

**Message forgery.** Syslog provides no authentication or origin verification. Receivers SHOULD use network-layer controls to limit acceptable message sources.

**Sensitive data.** Applications MUST NOT include secrets, passwords, cryptographic keys, or personally identifiable information (PII) in syslog messages.

**Denial of service.** An attacker may flood a syslog collector. Receivers SHOULD implement rate limiting and storage quotas.

## 10. IANA Considerations

IANA has created two registries for this specification.

**syslog Version Values.** IANA has registered VERSION = `1` as defined in this document (§6.2.2). Version numbers MUST be incremented for any new syslog protocol specification that changes any part of the HEADER. New values must be registered via the Standards Action method.

**syslog Structured Data ID Values.** IANA has registered the SD-IDs `timeQuality`, `origin`, and `meta`, together with their associated PARAM-NAMEs, as described in §7. New SD-ID and PARAM-NAME values must be registered through the IETF Review method. Once registered, syntax and semantics of SD-IDs and PARAM-NAMEs MUST NOT be altered; a change requires a new name. Names containing an at-sign (`@`) are locally extensible and are not controlled by IANA.
